文档协作工具的设计与实现
车思颖
西南大学计算机与信息科学学院 软件学院，重庆 400715
摘  要：互联网应用转向在线生产和多人协作后，传统单机文档编辑模式很难适应中小规模团队跨时空协作的要求，用户需要有协作、权限控制和版本追踪等功能的工具。本文为了解决该问题，设计并实现了一款面向中小规模协作场景的文档协作工具Together。系统使用前后端分离的架构，前端依靠Vue 3建立交互界面，后端利用Spring Boot和WebSocket处理业务逻辑及实时通信，通过MySQL进行数据持久化，主要实现了多人实时编辑、协作管理、版本追踪和文档导出等功能，采用WebSocket实时同步结合HTTP手动保存的方法平衡协作实时性和版本可控性，可以较好地满足中小规模团队协作需要，拥有实用价值和扩展空间。
关键词：在线文档协同编辑；WebSocket；Spring Boot；JWT



























Design and Implementation of a Document Collaboration Tool
CHE Siying 
School of Computer and Information Science and Software, Southwest University, Chongqing 400715, PR China
Abstract: 
After the Internet application turns to online production and multi-person collaboration, the traditional single-machine document editing mode is difficult to adapt to the requirements of small and medium-sized teams across time and space collaboration. Users need tools with collaboration, permission control and version tracking functions. In order to solve this problem, this paper designs and implements a document collaboration tool Together for small and medium-sized collaboration scenarios. The system uses a front-end and back-end separation architecture. The front-end relies on Vue 3 to establish an interactive interface. The back-end uses Spring Boot and WebSocket to process business logic and real-time communication, and uses MySQL for data persistence. It mainly implements functions such as multi-person real-time editing, collaborative management, version tracking, and document export. Using WebSocket real-time synchronization combined with HTTP manual preservation to balance collaborative real-time and version controllability can better meet the needs of small and medium-sized team collaboration, and has practical value and expansion space.
Key words: online document with collaborative editing; WebSocket; Spring Boot; JWT


第1章  导论
1.1 研究背景
近年来，文档的创作方式已经逐渐从传统的“单人离线编辑”转向了“多人在线协同”。无论是在企业办公、团队开发还是基层工作之类的场景当中，像项目策划方案、需求文档或者信息收集之类的工作，往往都需要由多名成员共同来完成。在这些任务里，文档所扮演的角色不只是内容的记录载体，它同时还承担着团队沟通、知识沉淀以及分工协作这些功能。过去的协作通常有两种做法：一种是通过即时通讯软件把文件反复传来传去，另一种是专门找人来对不同版本做人工合并。在项目刚开始或者团队规模还不大的时候，这种做法勉强还能应付；可一旦参与协作的人数变多，或者内容修改的频率变高了，版本冲突、内容被覆盖丢失、还有那些低效率的重复劳动之类的问题，就会越来越明显。
在深入观察团队实际协作过程后可以发现，目前的文档协同工作主要存在两点突出问题。第一点是效率问题：团队成员常常需要反复沟通，确认保留哪个版本，这种来回确认的过程会消耗很多时间和精力。第二点则体现在内容追溯上——当多人同时频繁修改同一份文档时，遇到某些关键内容被误删，想要迅速、准确地找回上一个稳定版本，有时操作起来比较困难。目前市面上成熟的大型在线协作文档平台，虽然功能上做得比较全面，但放到一些特定的使用场景比如中小型团队在敏捷开发过程中，或者有数据隐私保护要求、必须部署在私有化环境下的情况，这些平台往往显得有点笨重，功能冗余偏多、上手的门槛也比较高，无法适应轻量化需求。综合上述情况，设计一套功能更具针对性、架构更轻量、同时又比较容易进行定制拓展的文档协作平台，能够较好地贴合当前数字化办公以及敏捷开发模式下产生的实际需求。

1.2 研究目的
本文的核心目标是围绕实际使用场景，设计并实现一套可落地运行的在线文档协作系统 Together。具体目标如下：
1. 在功能上形成完整闭环，实现从用户注册登录到文档协同编辑、历史回滚、导出存档的全流程。
2. 在技术上采用主流、稳定、易维护的技术栈，保证系统能够在本科阶段可实现、可调试、可展示。
3. 在体验上尽量做到上手简单、操作直观，降低普通用户学习成本。
4. 在工程上保持清晰分层，为后续扩展评论、权限细化、差异对比等功能预留空间。

1.3 研究意义
本课题的意义可以从实践意义和学习意义两个角度说明。
在实践意义方面，Together 面向的是中小规模团队协作场景，强调“简洁明了、稳定、轻量”，可以真实服务于课程小组、毕业设计团队和小型项目组。对于这些场景，系统能明显减少文件来回传递带来的沟通成本，并提升多人写作效率。
在学习意义方面，本项目将前端开发、后端开发、数据库设计、网络通信和系统测试串成一个完整工程流程。相比单一功能练习，完整项目更能锻炼问题拆解能力和工程思维。特别是在实时协同场景中，需要同时考虑接口设计、消息同步、权限验证和异常处理，这对本科阶段综合能力提升非常有帮助。

1.4 国内外研究现状
在线协同编辑技术发展历史可以追溯到云计算和Web技术结合初期，在其发展历程中也不断有新的进展出现。从总体上看，该领域既有基础性研究也有应用型工作同时进行。而其中的代表作品如Google Docs、Microsoft 365等都是经过不断改进完善后才能拥有今天这样功能全面的服务。对于低级别的并发冲突处理问题，目前学者们以及企业界主要采取两种方案：一种是操作转换（Operational Transformation,OT）、另一种是无冲突复制数据类型（Conflict-Free Replicated Data Type, CRDT）。这两种方式都基于严格的数学模型以及分布式一致性理论来实现对多个用户同时编辑时的数据一致性。而且这二者都能很好地满足复杂的合作需求并且极大地提高了工作效率以及使用感受。
目前我国协同办公行业发展迅速，国内代表性产品如石墨文档、腾讯文档、飞书文档利用自身优势取得良好性能表现。基于云计算技术，它们搭建起一套完整的云上协同办公平台，在文档编辑基础上还加强了项目管理和跨部门合作等功能并且可以应用于手机等移动设备上。学者们对这一课题也进行大量相关研究工作，力求保障信息安全同时提高工作效率。比如，刘婷婷提出了一种针对智能制造行业的分布式协作方式，通过对数据集中统一管理和权限控制来达到多方高效协作的目的[3；李颜翎借鉴敏捷开发理念“看板”，设计了一个可视化的任务调度程序，大大加快软件开发速度[4]；而吕玉桂与范鑫怡则是以Spring Security框架以及JWT标准为基础实现微服务环境下身份验证和授权功能，同时解决Token失效以及角色变化等问题[5][8]。以上成果都说明分布式协同对于推动数字化经济发展起到关键作用并且未来很有可能成为构建智能知识服务平台的有效手段。
尽虽然OT协议以及CRDT等相关方法可以很好地解决分布式系统中的一致性问题，但是它们的工作原理以及基于事件的方法使得开发难度大大提高。一般都需要耗费大量的时间和精力去进行测试之后才能够稳定地工作。为了解决移动设备和云计算结合所带来的资源限制问题，研究人员也在寻求更好的协作编辑的方式。Lv等人提出一种基于CRDT理论实现的一种云-端协同编辑的方法，在此方法中将大量的计算移到服务器端完成而客户端只需要简单的操作就可以达到较好的用户体验以及较低的能量消耗[2]。这说明，“云上集中处理，靠近用户的设备辅助”，这种方式是应对资源短缺的一种可行思路。
本研究兼顾学术理论与工程应用的需求，在设计上充分考虑到工业级一致性管理系统所面临的复杂度以及不同场景下具体应用的区别。如果想要开发一个可以进行完整状态转移的功能型管理系统，那么这将是非常困难而且很难做到轻量化。因此本文从系统的复杂性和合作效率方面考虑，给出一种兼顾灵活性和高效性的方法，在通信层面使用WebSocket来保证低延时的通信，在存储上使用显式的存储加上增量快照的方式来进行版本控制以便于追踪文档的操作历史并能够方便地回滚到之前的状态。相比于传统的分布式的共识算法，这种方法去除了不必要的部分，既保证重要的合作能力，也做到了低成本、高可靠性和良好的用户体验。

第2章  系统相关技术介绍
2.1 前端技术栈
共同编辑系统的表现层使用Vue 3模式来完成建立工作。Vue 3所包括的组件化架构和组合式API设计可以提高细粒度视图的解耦程度、代码复用比例，并且也能够处理页面视图相对独立但内部交互逻辑比较密集的复杂工程场景。系统前端按照业务领域划分为身份认证、包括登录和注册、用户工作台Dashboard、富文本协作编辑等主要视图，这样做使系统的模块化程度和可维护性得到提高。
Together在路由和状态管理模式上，利用Vue Router来对前端路由生命周期进行控制，并且实现以角色为基础的页面级访问守卫。本文把Pinia当作轻量级的全局状态容器，用来集中调度被多个组件共享的上层应用状态，这里面包括了会话信息、当前激活文档和协作连接状态等，这样做能避免深部组件树之间出现复杂的属性透传问题。网络请求底层架构利用Axios库进行建立，通过设置请求拦截器来完成JWT凭证的自动注入，从而形成一条高内聚的HTTP通信链路，以此来减少重复性的样板代码。

2.2 后端技术栈
系统服务端使用Spring Boot框架体系构建RESTful API服务架构。工程代码严格遵循领域驱动与MVC分层思想，将系统整体划分为控制层（Controller）、服务层（Service）与数据访问层（Repository）：控制层负责对外部HTTP协议进行适配并做好请求派发的工作；服务层负责封装核心业务处理逻辑，同时对事务边界进行控制；数据访问层则专注于底层数据的持久化抽象操作。这种经典的纵向分层模型能有效实现系统解耦，从而保障业务逻辑的清晰性、独立性，以此为后续迭代升级工作保驾护航。
为了构建系统安全防线，Together系统深度集成Spring Security框架以实现接口级别的鉴权访问。针对资源访问动作，后端通过定制化的安全过滤链统一拦截并校验Token签名从而做到把横切关注点从具体的业务控制层中有效分离出去。已有研究显示Spring Security结合JWT的无状态鉴权方案能够有效满足分布式环境下的身份验证需求，且通过Redis缓存Token和动态刷新机制可以进一步提升系统的安全性与用户体验[5][6]。系统还引入全局异常处理机制，将底层服务抛出的各类运行时异常封装为标准化的JSON响应结构，保障前后端分离架构下接口契约的一致性，有效提升团队合作效能。

2.3 实时通信技术
在多人文档协作场景下，“单人输入、全局实时同步”是平台的核心需求。传统基于HTTP协议的请求-响应（Request-Response）半双工模式受限于无状态和单向通信的特性而难以应付高频的状态同步任务；而传统的定时轮询方案会引发高昂的网络开销，对服务器的负载也较大。基于上述情况，本系统在实时通信链路中引入WebSocket协议并建立客户端与服务端之间的全双工长连接通道，从而赋予服务端面向浏览器的消息主动推送、高效处理的能力。
把该情况放到并发编辑场景里进行分析，各个客户端会不断利用WebSocket信息通道来发送增量变更操作，服务端接收到这些信息后，按照文档唯一标识在内存路由池里准确查找，并且把变更指令以异步多播的形式分发给正在在线协作文档中的成员，以此达到在同一文档视图上实现实时状态同步的目的。WebSocket推送模型对比传统轮询机制，在控制通信延迟和提高带宽资源利用率上表现出明显优势。Murley等人对Tranco Top 1M网站进行了测量[1]，结果表明HTTP轮询的平均请求头大小为184.9字节，响应头为403.1字节，WebSocket数据帧的头部通常只有2至14字节，并且不用为每条消息重复发送HTTP头部。分析结果也表明，在在线聊天这类实时性要求比较高的场景中，WebSocket可以把单条消息的网络开销降低约80%以上[1]。陈诗雨等人[7]提出了一种使用WebSocket的共享同步控制方法SSCM，该方法通过定义在线编辑用户队列、正在编辑用户表，使多用户能够对文档段落进行互斥访问，并且实现状态转换。孙瑞瑞[11]使用WebSocket和操作转换技术建立了面向科研团队的文档协同编辑平台。上述案例为Together系统实现高频次和低卡顿的共同编辑功能提供了可靠的网络理论基础支持。。

2.4 数据存储与鉴权技术
在数据持久化层面，系统选用MySQL数据库作为核心的数据存取引擎。平台所需的存储的业务实体主要包括用户身份信息、文档元数据、多对多协作关系映射以及核心版本追踪记录等等。得益于关系型数据库在数据一致性方面的严谨性与规范化设计思路，Together系统能够把“单个用户拥有多篇文档”、“单篇文档关联了多个协作者”这类比较复杂的实体依赖关系给准确地表达出来并维护好，同时借助其事务机制来保障并发写场景下数据的完整性。
在服务授权机制层面，本系统采用一种基于JSON Web Token（JWT）的无状态鉴权方案。在用户通过认证后，服务端会用特定加密算法签发JWT令牌，之后客户端发起的每一次受限资源请求，都需要在头部（Header）携带这个令牌。服务端只需要通过解析签名即可完成用户身份识别与权限放行。这种模式不仅能消除传统Session机制对服务端内存存储的依赖，而且因为它本身具有的自包含特性与现代前后端分离架构需求非常契合。针对JWT在实际使用过程中会遇到的的Token过期、无法主动失效等问题，已经有研究提出过结合Redis服务端存储和动态刷新机制的改进方案[5][6]，Together系统在设计时也借鉴了其中部分思路，最终通过手动保存版本以及快照机制，辅助实现了可控的会话管理。

3.1 可行性分析

3.1.1 技术可行性
Together系统依赖的核心技术框架——前端Vue 3、后端Spring Boot、数据库MySQL以及全双工通信协议WebSocket，都属于当前工业界高度成熟且被广泛应用验证的技术体系。Vue 3的响应式系统和虚拟DOM机制能够高效应对交互密集型应用中频繁的DOM更新操作；Spring Boot强大的生态隔离和敏捷开发特性，为构建高内聚的RESTful接口提供了便利；而WebSocket则可以从协议底层保证多路复用情况下的低延迟消息投递。况且上述技术都拥有庞大的开源社区支持和详细的官方文档，完全能够支撑起一个实时协同编辑系统的平稳落地。本系统还简化了OT算法中复杂的操作转换逻辑，仅保留其“操作同步与状态一致”的核心目标，进一步降低了技术实现难度、提升了项目的技术可行性[1][2]。

3.1.2 经济可行性
Together项目采用的框架链、数据库引擎以及相关中间件都是遵守开源协议的免费软件，从而极力避免高昂的商业软件授权费用。在硬件与部署层面，系统的架构遵循轻量级与模块化设计原则，不管是前期的本地物理机开发调试、还是后期的系统部署，都不依赖昂贵的高性能计算集群或特殊硬件设备。故而Together从研发、测试直到最终上线的整个生命周期内产生的经济开销非常低，具备良好的经济可行性。


3.1.3 操作可行性	
在用户体验与人机交互设计上，Together项目秉持极简主义的设计理念。界面没有传统的菜单层级，而是提炼出一条“身份认证—进入工作台—创建/定位文档—沉浸式协同编辑”的线性操作路径。界面元素与交互逻辑符合现代互联网用户的使用习惯，用户可在短时间内熟练掌握系统的核心功能。

3.2 用户角色与业务场景分析
为了理清系统边界和数据流向，Together系统将平台上的用户划分为两种类型：文档所有者与协作成员。文档所有者掌握文档的整个生命周期，他们有权执行一些高权限操作如邀请成员加入、删除文档等。协作成员则是通过邀请进入文档空间，他们能够在线编辑、实时协同并查阅内容。通过这样划分角色，系统能够分离“管理权限”与“编辑权限”，从而减少越权风险。

3.3.1 业务需求分析
Together系统需要满足以下需求：
提升团队协作效率：解决传统离线编辑中的版本混乱与文件传输延迟等问题，实现多人异地实时协同，降低沟通成本。
保障数据资产安全：建立比较严格的RBAC权限体系和历史版本追溯机制，防止有人误删或恶意篡改文档内容。
降低使用与维护门槛：面向中小规模团队提供轻量级服务，避免有些商业软件存在的功能冗余，实现平衡低成本部署与易用性。

3.3.2 用户需求分析
文档拥有者作为文档的创建者和管理者，具备完整的文档管理权限。文档创建功能允许文档拥有者创建新的协作文档，初始化文档元数据和权限设置，验证用户身份和权限，生成唯一文档ID，创建文档元数据记录，初始化文档内容存储，建立文档-用户所有权关系，创建初始版本记录，返回文档访问信息。
查看文档列表功能在用户登录后的工作台（Dashboard）直接展示当前用户关联的所有文档，并明确标识用户在每个文档中的角色（如 Owner 或 Editor）。
编辑文档内容功能允许文档拥有者修改文档内容，更新文档内容状态。删除文档功能允许文档拥有者永久移除文档内容记录及相关协作关系。
协作管理功能主要为邀请协作者和处理邀请。文档拥有者能够邀请其他用户加入协作，先触发系统验证操作者文档所有权，然后系统查询目标用户是否存在并检查是否已是协作者，检测之后系统给对应用户发送邀请通知，建立协作关系，被邀请的用户有权接受或拒绝协作邀请。
版本控制功能包括手动保存版本、查看历史版本列表和版本回滚。用户能通过保存按钮手动创建文档历史版本，捕获当前文档的所有内容并生成版本快照。前端页面将显示历史版本的基础信息包括版本号、保存时间、操作人。用户可直接选择列表中的某一历史版本进行一键回滚操作，系统将用该版本的快照内容覆盖当前文档内容。



3.3.3 系统功能需求分析
基于上述用户需求，将其转化为系统必须实现的具体功能模块。系统功能需求分为通用功能、文档管理功能、协同编辑功能和版本管理功能四大类。
1.用户认证与管理功能
FR-01 用户注册：前端提供注册界面，系统校验邮箱格式唯一性并使用BCrypt算法加密存储密码。
FR-02 用户登录：系统验证用户凭证，生成JWT Token并返回给前端，之后记录登录日志。
FR-03 会话维持与退出：前端在请求头中自动携带Token，后端拦截器验证Token有效性后予以放行，若Token无效则直接拦截。
2.文档全生命周期管理功能
FR-04 文档创建：用户点击创建新文档时，系统初始化文档元数据（标题、UUID、所有者ID）并在数据库中建立所有者关联关系。
FR-05 文档列表查询：系统根据当前用户ID关联查询documents和document_members表，后端将获取到的信息发送给前端，前端渲染展示文档信息，包含标题和角色权限标识。
FR-06 文档基础操作：文档所有者能对文档进行删除，执行前系统将进行权限校验。
FR-07 文档导出：系统为用户提供文档导出能力，支持将文档内容转换为HTML或PDF等常用格式供用户下载。
3.协作邀请与权限控制功能
FR-08 发起邀请：文档所有者可通过输入协作者用户名发起邀请，系统在document_invitations表中生成状态为PENDING的记录。
FR-09 处理邀请：被邀请用户可在个人中心查看邀请列表，执行接受或拒绝操作。若接受，系统在document_members表中插入记录并更新邀请状态为ACCEPTED。
FR-10 权限校验中间件：在所有文档操作接口前置权限校验逻辑，根据document_members中的角色字段（OWNER/EDITOR）判断是否允许执行相应操作，非法请求返回403 Forbidden。
4.实时协同编辑功能
FR-11 WebSocket连接建立：系统提供WebSocket端点以支持客户端建立长连接，并根据Document ID订阅对应的消息频道。
FR-12 实时内容同步与并发控制：
多用户并发编辑支持：多名拥有编辑权限的用户可以同时打开并修改同一文档。任何用户的文本输入操作（增、删、改）都会通过WebSocket实时广播给其他所有在线协作者。
冲突处理策略：针对极端的并发修改冲突（如多用户在极短时间内修改同一字符位置），系统采用“最后写入优先（Last-Write-Wins，LWW）”策略，即服务端以接收到的时间戳为准，后到达的操作覆盖先前的操作状态，确保文档内容的最终一致性。
FR-13 在线状态感知：当有用户进入或离开文档页面时，系统通过WebSocket心跳机制或断开事件更新状态，然后广播给其他协作者。前端通过渲染不同颜色的高亮显示其他用户的编辑位置和编辑内容，让用户直观感知并发操作，从而减少逻辑冲突的发生。
FR-14 协作聊天：系统基于WebSocket搭建简易的聊天室功能，当前文档在线成员可以在聊天室发送文本消息进行沟通。
5. 历史版本与容灾功能
FR-15 手动保存版本：当用户点击保存版本按钮，系统会触发HTTP请求。将当前文档完整内容快照存入document_operations表，并记录版本号、操作人及时间戳。
FR-16 历史版本列表：按时间倒序展示文档的所有历史版本快照，并显示版本号、保存时间及操作者等信息。
FR-17 版本回滚：用户可以选择某一历史版本进行回滚，系统将对应快照内容覆盖当前文档内容，并记录一次“回滚”操作日志。
FR-18 冲突后的数据恢复（兜底机制）：
系统需要拥有手动版本快照功能，以避免LWW策略让重要数据发生意外覆盖，见FR-15。用户察觉到实时同步引发的内容错乱或误覆盖后，能够利用版本回滚，见FR-17，将文档恢复至冲突出现前的稳定状态。

3.4 非功能需求分析
除了满足业务逻辑闭环，系统还需要遵守下面的非功能性约束：
实时性和低延迟：在网络环境正常的情况下，协同编辑操作指令的端到端同步延迟要控制在数百毫秒以内的极低水平，使用户在编辑过程中输入的内容可以实时同步给所有在线协作者，不会出现明显的操作滞后感。
并发稳定性和高可用性：服务端需要拥有异常捕获和并发隔离的能力，当多用户高频多点并发输入造成瞬时数据洪峰时，不会出现系统崩溃。
严格的隐私隔离方式：系统建立起全流程的纵深防御模式，用来防范用户操作中可能产生的纵向越权和横向越权访问行为。通过使用权限管控和数据隔离等多种方式，用户可以协同创作生成的文档数据内容保持安全隐蔽，并且达到不同用户及不同团队之间的完全隔离，以此来防止数据泄露的风险。
架构具有扩展性和可维护性：系统建立遵守RESTful架构风格和前后端分离的约束，中心代码结构清楚，达到模块之间低耦合和高内聚的状态。
网络容错和状态恢复：用户在使用过程中如果遇到弱网环境或因网络抖动而短暂连接中断，客户端应该执行断线重连操作。重连成功后，系统要保证用户主要协作状态的一致性，达到编辑操作和在线状态等内容的顺畅衔接，不会因为短暂的网络异常造成用户编辑数据丢失。

3.5 数据需求分析	
文档协作系统的主要资产为数据，业务逻辑流转需要数据作为载体。为保证系统功能可以正确运行、数据保持一致并且用户信息受到保护，本文把前文所述的功能需求和角色定义结合起来，在主要数据实体、数据完整性限制、数据一致性方式和数据安全隐私这四个方面，分析了系统的数据需求。
3.5.1 核心数据实体需求
结合前文的分析，可以进一步梳理出系统需要持久化存储的五类核心业务实体。
用户身份实体：它负责存储系统里所有注册用户的基础身份信息，是实现权限控制与会话管理的基础。关键属性有：用户唯一标识、登录邮箱、用户名、加密后的密码哈希、显示名称，以及账户创建时间。其中，邮箱和用户名在全局范围内不能重复；密码字段则必须经过加盐哈希处理，保障安全性。
文档核心资产实体：这是协同编辑的主要对象，负责存储的是在线文档的元数据以及当前最新的内容快照。关键属性有：文档唯一标识、标题、正文内容、所有者ID、当前版本号，以及创建和更新时间。注意这里的版本号字段能防止多个用户同时保存时，后保存的人把前面人的改动给覆盖掉。
协作关系实体：它是实现RBAC（基于角色的访问控制模型）的关键数据支撑，负责记录用户和文档之间的多对多关联关系，以及具体的权限级别。为了防止重复授权，这里添加了一个联合唯一约束。该实体中的角色字段，直接决定用户在某个文档里能做什么、不能做什么。
邀请流程实体：这个实体用来记录文档所有者发起的协作邀请，以及邀请的流转状态，支撑异步的协作加入流程。它记录从“发起邀请”到“建立协作关系”中间的状态机变化过程，从而确保协作权限的授予是可追溯的、安全的。
历史版本实体：这个实体是为“手动封板快照模型”提供数据底座的，支持版本回溯和容灾恢复。它的触发机制是手动的——每次保存都会生成一条独立的记录。关键要求是快照内容必须完整，这样当出现误操作或者数据损坏时，能精确回滚到任意一个历史节点。

3.5.2 数据完整性与约束需求
系统数据应当符合实体完整性、参照完整性和域完整性这三种约束，从而保证数据质量和业务逻辑的严谨性。所有主要数据表都应定义主键并且主键值不能为空和唯一，比如users表里的email和username字段要设置唯一索引，在数据库方面上可以避免重复注册情况，document_members表要对document_id和user_id建立联合唯一约束，以免同一个用户对同一个文档生成多条权限记录。外键关联应当保持有效并且一致。例如，documents表中的owner_id需要指向users表中真实存在的用户ID，而且document_operations表中的document_id需要关联有效的文档记录。系统在进行删除操作时，应当按照业务规则来制定级联策略，这样可以防止出现孤儿数据。数据字段要符合特定格式和类型及范围约束，比如邮箱字段要符合标准正则表达式格式，时间戳字段要自动获取服务器当前时间。

3.5.3 数据一致性需求
在实时协同编辑的业务流程里，系统会允许存在毫秒级的多端短暂状态不同步，该方式同时兼顾了协同操作的流畅性和同步效率。系统利用WebSocket全双工长连接的广播方式，在网络环境稳定的情况下，把单用户的编辑操作实时同步到全部在线协作者的客户端，最终保证所有终端展示的文档内容达到完全一致。系统在极端场景下处理并发修改冲突时会采用最后写入优先的方式进行冲突消解，并且利用用户手动保存的版本快照来确定数据状态，从而为内容一致性提供最终的保障。
用户在进行重要业务操作时，系统要严格保证数据库操作的事务一致性，主要依靠实现事务的原子性特点，来确保关联操作形成完整的闭环。就用户接受协作邀请并建立正式协作关系的业务流程而言，该操作包括两个强关联的数据库执行步骤：第一步是对document_invitations表中对应邀请记录的状态进行更新，第二步则是把该用户和对应文档的权限关联记录插入到document_members表中。这两个步骤应当被安排在同一个数据库事务中进行执行，遵循只有全部执行成功或者全部执行失败才会进行回滚的原子性原则，这样做是为了防止邀请状态已经更新但是用户协作权限没有生效的异常情况，同时也能够避免数据脏读问题的出现。用户进行的保存版本操作，也需要在同一个事务里完成两项相关操作，即同步更新文档对应的当前版本号，并且把对应的版本快照记录插入到历史版本表中，这样做是为了保证文档版本迭代链路的完整和连续，防止出现版本链断裂、版本号与快照不匹配这类异常问题。

3.5.4 数据安全与隐私需求
文档包括敏感信息，数据需求也包括敏感数据加密、访问隔离和权限过滤，还有数据备份和恢复方式，用来防止数据被轻易泄露。用户密码不可以以明文形式保存，需要利用哈希算法并加入随机数来保存，对于将来有可能扩展的敏感字段，也应该运用对称加密算法来保存密文。数据查询层应当执行严格的权限过滤。比如查询本人参与的文档列表时，SQL语句要关联document_members表并限制当前用户ID，这样做可以防止通过接口遍历获取他人私有文档ID或内容，达到行级数据隔离的目的。系统应当拥有定期备份主要数据的功能，并依靠历史版本实体，当出现严重逻辑错误或者数据污染时，可以利用快照进行手动恢复，从而给数据安全建立最后一道防线。


第4章  系统总体设计
4.1 系统架构设计
Together系统采用前后端分离的设计范式进行架构。客户端环境（前端）主要负责视图渲染、人机交互响应以及复杂协作状态的维护；服务端应用（后端）则主要承担核心业务逻辑路由、安全鉴权、大规模数据持久化以及全双工消息的多播投递。不同端点之间通过遵循HTTP接口与WebSocket协议，实现松耦合的数据交换过程。
从软件工程控制复杂度的分层视角审视，系统整体可划分为以下四个核心层次：
表现层（Presentation Layer）：表现层由Vue 3组件树构成，主要负责解析用户输入事件、实时渲染呈现文档数据以及构建沉浸式的协同编辑器视图。
接入层（Interface/Gateway Layer）：接入层拥有双通道特点，HTTP接口对外提供标准化的API，并且对常见的增删改查等单工请求进行处理。WebSocket持久连接部分充当消息网关，主要负责进行高频率的实时状态同步。
业务逻辑层（Business Logic Layer）：业务逻辑层建立在后端的Service组件上，集中封装权限校验、文档流转规则、一致性维护以及全局异常捕获等核心机制。
数据持久层（Data Persistence Layer）：数据持久层依靠MySQL关系型存储引擎，对系统中的用户实体、文档快照、权限拓扑等核心要素进行存储，保障事务级的并发写入。

4.2 功能模块设计
M1. 身份与边界控制模块（Auth & Security Domain）
身份与边界控制模块是系统的统一准入防线，负责实现用户的注册与安全登录逻辑。
M1.1 用户认证子模块：
用户认证子模块使用无状态的JWT颁发和校验方式，来完成安全网关的工作。在Controller链路前端进行集成，以实现对Spring Security定制过滤器链的使用，并对非法请求进行拦截。本文利用BCrypt哈希算法对密码进行存储，以此来保护用户敏感信息的安全。
M1.2 会话管理子模块：
会话管理子模块对HTTP请求头中的Authorization字段进行解析，从中提取用户ID和角色信息，并且把这些数据注入到当前线程上下文里，供其他业务模块使用。

M2. 数字资产与空间管理模块（Document Lifecycle Domain）
M2.1 文档CRUD子模块：
文档创建时，文档CRUD子模块提取用户载荷并生成唯一的UUID与初始时间戳，同时建立该用户为OWNER的document_member外键映射，实现建表与确权；根据获取的Token中的身份UID信息，通过关联查询拉取该与该用户相关联的文档列表；提供对文档标题和内容的增量更新接口，支持LONGTEXT类型的大文本存储，实现内容持久化。
M2.2 协作邀请子模块：
协作邀请子模块包含一套基于邀请票据机制驱动的协作者流转闭环，文档所有者可以发起定向邀请，生成带有status（PENDING/ACCEPTED/REJECTED）状态机的邀请记录。被邀请用户通过接受邀请接口触发document_members表的插入操作，从而获得文档访问权限。


M3. 细粒度权限管理模块（Permission Control Domain）
服务端对细粒度权限管理进行严格管理，依靠执行基于角色的访问控制RBAC来实现，横向越权访问在底层逻辑中受到隔离。
M3.1 角色定义与映射：
系统定义两种核心角色：OWNER、EDITOR，权限关系存储在document_members表中，通过role字段进行枚举标识。
M3.2 动态权限校验器：
在执行删除文档、邀请成员、回滚版本等敏感操作前，Service层会调用权限校验工具类，校验逻辑是：先判断当前用户是不是文档所有者（Owner）；如果不是，则查询document_members表获取其在该文档下的角色，再比对操作所需的最小权限等级，若权限不足，直接抛出AccessDeniedException，并由全局异常处理器返回403 Forbidden。

M4. 全双工高频协同引擎（Real-time Collab Engine）
M4.1 WebSocket 消息网关：
消息网管利用独立构建的WebSocket管道，承载文档草稿内容和版本信息的高频上传及扇出广播，根据Document ID将消息推送到对应的STOMP主题频道如/topic/document.{id}，保证在线端能快速完成状态同步。
M4.2 在线状态感知子模块：
在线状态感知子模块集成了心跳探测功能，并且会动态维护一个活跃成员池。用户在进入或离开编辑页面时，通过WebSocket发送Presence事件，服务端广播最新的在线用户列表，以达到实时同步的目的。
M4.3 并发协调与冲突解决子模块：
客户端为每个操作生成唯一的时间戳和客户端ID。服务端收到消息直接根据Document ID进行广播，如果出现物理位置冲突（比如两人同时改同一个字），以后到达服务端的消息为准。由于这么做可能会导致数据覆盖，系统还依赖M5版本溯源模块。每次手动保存生成的快照用于极端情况下的数据恢复。


M5. 版本溯源与历史管理模块（Version History & Tracing Domain）
M5.1 版本快照生成子模块：
触发机制采用手动封板快照模型，当用户点击“保存版本”按钮时，触发HTTP请求。利用JPA的@Transactional事务保障原子性，将当前文档内容复制到document_operations表中。元数据记录同时保存base_version（基准版本号）、target_version（衍生终态版本）、operator_id（操作人ID）及timestamp（时间戳），形成不可篡改的历史链条。
M5.2 历史查询与对比子模块：
提供分页查询接口，按时间倒序返回指定文档的历史版本列表并交给前端进行渲染。
M5.3 版本回滚子模块：
系统在接收到回滚请求后，会把历史快照反写回当前的documents表中，并且会抛弃被污染的时间线切点。在进行回滚写入操作时，系统会对文档当前的version字段和请求发起时的版本进行比较，这样可以防止在回滚过程中其他用户提交了新内容，从而避免脏写发生。如果出现版本冲突，系统会拒绝进行回滚操作，并且提示用户对系统进行刷新。

4.3 数据库设计（E-R模型抽象）
users（平台用户表）
users表作为业务操作的最高实体层，抽象并记录平台的物理账号数据，其关键字段做了防碰撞设计：登录邮箱（email）与标识名（username）具备严格唯一性（Unique），不会重复；密码数据（password_hash）经过加密算法处理，不以明文形式存储。附加display_name实现对外视觉层代换，而created_at等常规审计字段则用于监控生命周期之源。
documents（文档核心资产表）
documents表表承载了系统的核心内容，内部不仅存放了长文本数据，并在外部通过doc_uuid实现了不连续的安全外部ID，可在一定程度上防止爬虫遍历。documents表配置了version和时间戳用于乐观锁控制。该表通过外部约束外键owner_id指向唯一的文档创建者，形成稳定的主从依赖关系。
document_members（权限成员关系表）
因为一个文档会涉及到多位协作者，同时一个用户也可以参与多个文档，所以抽象出document_members表来解决多对多（M:N）的映射问题。表中把document_id和user_id组合起来设置为联合唯一约束，防止重复加入，其中的枚举字段role（取值如OWNER、EDITOR）为不同协作者分配不同级别的权限，同时通过joined_at记录加入时间。
document_invitations（邀请制流程状态表）
document_invitations表是一张独立派生记录表，用来串联从发起邀请到最终处理的完整时序流程。表中包含邀请发起方（inviter_id）与接收靶向方（invitee_id），表中特别设定了状态机推演字段status如PENDING（待办），ACCEPTED（接纳），REJECTED（驳回）和授予角色预期role，较好地还原了异步状态审批的过程，让团队接入权限的追溯更加清晰。
document_operations（操作审计与快照表）
document_operations表是对传统粗放式全量文档备份的一种工程优化。相比之前每次保存都备份整篇文档的做法，该表只记录每次版本变更的情况：通过base_version（基准版本号）和target_version（目标版本号）进行双向定位，同时用长文本字段op_payload（本次提交的内容）和content_snapshot（快照字段）来保留可回溯的数据。表中还加入了唯一的client_op_id用于请求防重，以及事件类型枚举，可以用于冲突排查和定点回滚等场景。

5.1 安全认证与会话管理实现
本章将围绕Together系统的几个关键功能模块的具体实现过程展开说明。
5.1.1 认证流程设计
认证流程分为注册、登录与令牌验证三个环节：
注册：前端把用户填写的信息提交给后端，后端收到后用BCrypt算法对密码做加密处理，再把加密后的密文和其他用户信息一起写入到users表中。
登录：用户输入账号和密码后提交，后端先校验账号密码是否匹配。如果校验通过，后端会生成一个JWT，这个Token里面包含了用户ID、角色信息以及过期时间，然后将其返回给前端。
鉴权拦截：前端在Axios的请求拦截器里做配置，让每次发出的请求都自动带上JWT。后端这边则通过JwtAuthenticationFilter来解析Token，如果解析成功且合法，就构建出一个Authentication对象并将其存入 SecurityContext中；如果解析失败或Token不合法，就直接抛出异常。

5.1.2 关键代码实现
1.JWT工具类实现
JwtUtil负责Token的生成与解析，确保签名的安全性。
@Component
public class JwtUtil {
    @Value("${jwt.secret}")
    private String secretKey;
    @Value("${jwt.expiration}")
    private long expirationTime;

    /**
     * 生成 JWT Token
     */
    public String generateToken(UserDetails userDetails) {
        Map<String, Object> claims = new HashMap<>();
        // 自定义 Claims，如加入用户ID
        claims.put("userId", ((CustomUserDetails) userDetails).getId());
        return Jwts.builder()
                .setClaims(claims)
                .setSubject(userDetails.getUsername())
                .setIssuedAt(new Date())
                .setExpiration(new Date(System.currentTimeMillis() + expirationTime))
                .signWith(SignatureAlgorithm.HS256, secretKey.getBytes(StandardCharsets.UTF_8))
                .compact();
    }

    /**
     * 验证 Token 有效性
     */
    public boolean validateToken(String token, UserDetails userDetails) {
        final String username = extractUsername(token);
        return (username.equals(userDetails.getUsername()) && !isTokenExpired(token));
    }
}

2.安全过滤链配置
在SecurityConfig中配置过滤链，放行登录注册接口，拦截其他受保护资源。
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Autowired
    private JwtAuthenticationFilter jwtAuthFilter;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // 前后端分离通常禁用 CSRF，依赖 JWT
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll() // 放行认证接口
                .requestMatchers("/ws/**").permitAll()       // 放行 WebSocket 握手
                .anyRequest().authenticated()
            )
            .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);
        
        return http.build();
    }
}

// src/utils/request.js
import axios from 'axios';
import { useAuthStore } from '@/stores/auth';

const service = axios.create({ baseURL: '/api' });

service.interceptors.request.use(config => {
    const authStore = useAuthStore();
    if (authStore.token) {
        config.headers['Authorization'] = `Bearer ${authStore.token}`;
    }
    return config;
});

service.interceptors.response.use(
    response => response.data,
    error => {
        if (error.response.status === 401) {
            // Token 过期或无效，跳转登录页
            const authStore = useAuthStore();
            authStore.logout();
            window.location.href = '/login';
        }
        return Promise.reject(error);
    }
);

export default service;

5.2 文档管理与控制模块实现
文档管理模块负责文档元数据的CRUD操作及协作成员的管理。该模块严格遵循RBAC模型，确保只有拥有相应权限的用户才能执行特定操作。

5.2.1 核心业务逻辑
1. 文档创建与所有权确立 
当用户创建文档时，系统需原子性地完成两个操作：
（1）在documents表中插入文档记录。
（2）在document_members表中建立当前用户与该文档的关联，并赋予OWNER角色。
@Service
@Transactional
public class DocumentService {
    @Autowired
    private DocumentRepository documentRepository;
    @Autowired
    private DocumentMemberRepository memberRepository;

    public Document createDocument(CreateDocumentRequest request, Long userId) {
        // 1. 创建文档实体
        Document doc = new Document();
        doc.setTitle(request.getTitle());
        doc.setContent(""); // 初始内容为空
        doc.setOwnerId(userId);
        doc.setDocUuid(UUID.randomUUID().toString());
        Document savedDoc = documentRepository.save(doc);

        // 2. 建立所有者关系
        DocumentMember ownerRelation = new DocumentMember();
        ownerRelation.setDocument(savedDoc);
        ownerRelation.setUser(new User(userId)); // 简化写法，实际需查询用户
        ownerRelation.setRole(MemberRole.OWNER);
        memberRepository.save(ownerRelation);

        return savedDoc;
    }
}

2.权限校验切面 
为了防止越权访问，系统在Service层或Controller层引入权限校验逻辑。以删除文档为例，仅允许Owner执行。
public void deleteDocument(Long documentId, Long currentUserId) {
    Document doc = documentRepository.findById(documentId)
            .orElseThrow(() -> new ResourceNotFoundException("Document not found"));
    
    // 校验当前用户是否为Owner
    if (!doc.getOwnerId().equals(currentUserId)) {
        throw new AccessDeniedException("Only the owner can delete this document");
    }
    
    documentRepository.delete(doc);
}

5.2.2 协作邀请机制实现
邀请功能通过document_invitations表实现异步审批流。
发起邀请：Owner输入协作者邮箱，后端生成一条状态为PENDING的记录。
接受邀请：被邀请者登录后查看邀请列表，点击接受后，后端事务性地执行：
更新邀请状态为ACCEPTED。

@Transactional
public void acceptInvitation(Long invitationId, Long userId) {
    Invitation invitation = invitationRepository.findById(invitationId)
            .orElseThrow(() -> new RuntimeException("Invalid invitation"));
    
    if (!invitation.getInviteeId().equals(userId)) {
        throw new AccessDeniedException("Not your invitation");
    }

    member.setDocument(invitation.getDocument());
    member.setUser(new User(userId));
    member.setRole(invitation.getRole()); // 使用邀请时指定的角色

}

5.3.3 冲突处理策略
由于未引入复杂的OT/CRD 算法，系统采用“最后写入优先（Last-Write-Wins，LWW）”配合“手动保存快照”的策略来解决并发冲突。
1.LWW 策略实现 
在实时阶段，所有用户的keystroke事件经过防抖处理后发送给服务端。服务端在CollabWebSocketController中为每条消息打上服务器接收时间戳serverTimestamp。当多个用户同时修改同一位置时，服务端按接收顺序广播，客户端以最后收到的消息为准更新视图。
// 在Controller中设置时间戳，确保全局时序一致
message.setTimestamp(System.currentTimeMillis());
2.兜底机制
针对手动保存快照这一机制，需要说明的是：由于LWW策略在极端并发情况下可能会出现数据覆盖的问题（比如两个人同时删除同一个段落里不同的部分，就可能互相影响），本系统提供了一个“手动保存”功能，作为保障数据安全的最后一道防线。
触发方式：用户点击界面上的“保存版本”按钮即可触发。
执行过程：后端收到请求后，会取出当前文档的完整内容，以快照的形式存入到document_operations表中。
恢复方式：如果用户发现因为实时同步导致文档内容出现了混乱或错乱，可以借助历史版本回滚功能，把文档恢复到冲突发生之前那个相对稳定的状态。

5.4 历史追溯与快照容灾实现
历史版本模块旨在保障数据安全性，允许用户查看过往状态并回滚。

5.4.1 版本快照生成
当用户点击“保存版本”时，前端发起 POST 请求。后端执行以下逻辑：
@Service
public class HistoryService {
    
    @Autowired
    private DocumentRepository documentRepository;
    @Autowired
    private DocumentOperationRepository operationRepository;

    @Transactional
    public void saveVersion(Long documentId, Long userId, String comment) {
        Document doc = documentRepository.findById(documentId)
                .orElseThrow(() -> new ResourceNotFoundException("Document not found"));
        
        // 创建操作记录/版本快照
        DocumentOperation op = new DocumentOperation();
        op.setDocument(doc);
        op.setUser(new User(userId));
        op.setOperationType(OperationType.MANUAL_SAVE);
        op.setContentSnapshot(doc.getContent()); // 保存当前全文快照
        op.setBaseVersion(doc.getVersion());
        op.setTargetVersion(doc.getVersion() + 1);
        op.setComment(comment);
        op.setTimestamp(LocalDateTime.now());
        
        operationRepository.save(op);
        
        // 更新文档当前版本号
        doc.setVersion(doc.getVersion() + 1);
        doc.setUpdatedAt(LocalDateTime.now());
        documentRepository.save(doc);
    }
}

5.4.2 版本回滚实现
回滚操作本质是将指定历史版本的快照内容写回当前文档。
@Transactional
public void rollbackToVersion(Long documentId, Long operationId, Long userId) {
    // 1. 获取目标历史版本
    DocumentOperation targetOp = operationRepository.findById(operationId)
            .orElseThrow(() -> new RuntimeException("Version not found"));
            
    // 2. 权限校验
    if (!targetOp.getDocument().getId().equals(documentId)) {
        throw new IllegalArgumentException("Version mismatch");
    }
    
    // 3. 获取当前文档并更新内容
    Document doc = documentRepository.findById(documentId).get();
    doc.setContent(targetOp.getContentSnapshot());
    doc.setVersion(doc.getVersion() + 1); // 版本号递增，表示发生了新变更
    doc.setUpdatedAt(LocalDateTime.now());
    
    documentRepository.save(doc);
    
    // 4. 记录一次“回滚”操作日志，便于追溯
    DocumentOperation rollbackLog = new DocumentOperation();
    rollbackLog.setDocument(doc);
    rollbackLog.setUser(new User(userId));
    rollbackLog.setOperationType(OperationType.ROLLBACK);
    rollbackLog.setComment("Rolled back to version " + targetOp.getTargetVersion());
    operationRepository.save(rollbackLog);
}

7.1 本文的主要研究结论和成果
本文在在线云服务办公场景和远程学习常态化的大环境下，以学生和初创团队等中小规模协作中的实际问题为出发点，顺利完成了轻量级并发文档平台Together项目的主要技术论证、架构蓝图设计、全业务链编码实现、最终的鲁棒性验收测试工作。
本文和工程实际应用验证了以下主要结论：
建立高内聚轻型协作架构：避免了在中小团队中出现积重难返或被大厂垄断重载化的商业引擎，采用更符合敏捷开发模式的前后端分离架构、RESTful和长全双工WebSocket结合的技术方式，找到在项目复杂度和时间框限制下一条非常高的可用协作解决方案。
抽象设计安全权限模式：通过利用基于角色的纵向越权网关过滤和物理库方面上的UUID和时间版本锁概念，即乐观并发控制雏形，在极简的代码方面上提供并保证了包括资源创设、安全多维授权和定向共享编辑方式，有效遏制了数字资产外泄。完善灾难追溯和容错备份基座没有停留在粗放的内容全覆盖重写思维上，而是对基于用户指点的人工干预型基线快照模型进行了思考。通过原子级别的双版本事件保存，该系统在防篡改覆盖和防数据回退序列交叉这两个方面给出了确定的底层容灾回滚方式。

7.2 对后续规划和功能技术演进所进行的探索
软件开发是需求生命周期持续扩大的演替过程，由于单人工程实施的时效和硬件架构资源的限制，当前的初级单体工程在多条专业发展道路中仍然留有较大的架构升级空间，主要迭代方面可以具体规划为：
1.复杂并发冲突算法主要演进：目前轻量级的广播策略和客户端延迟打包模式会在密集并发中退化成粗暴覆写冲突，也就是Last-Write-Wins。系统未来在进行规模化承重时，需要剥离当前设计，引入主流的CRDT或OT的深层数学抽象模型。现有研究把CRDT应用到云和移动端共同编辑模式里，通过把操作转换、冲突处理迁移到云端，减少了移动端的计算和能耗开销[2]。采用该思路，系统可以在后端集群中使用CRDT或OT的主要算法，前端只进行已转换的操作，这样在维持轻量客户端的基础上，达到更严格的并发一致性。在权限控制上，可以参考基于信任的动态RBAC模式[6]，并把用户行为信任度评估方式加入进来，这样能进行细粒度且对上下文有感知的访问控制策略。
2.服务注册和集群消息背板的改造，分布式规模。WebSocket本地会话注册池在多物理节点负载均衡部署过程中，会出现广播信道被割裂的问题。后续工程部署应当引入Redis发布订阅模式或Kafka消息队列来建立分布式底层网关网络，从而可以支持文档路由的无限扩展和水平容器的动态增加。刘迎迎[9]提出了增量式同步方法，该方法利用消息队列对数据变化进行监控，并只同步差异部分，该方法可以降低系统负担。未来可以把该方案引入本系统的文档版本同步方式，用来替换当前的全量广播策略。
3.视图解析、精细化办公能力的分析：本文从用户角度出发对能力进行添加，具体包括引入底层文档节点AST渲染来达到精确的差异比对记录Diff-View，扩展细分类别的主被动历史批注控制Annotations，丰富文档版本导出的规范模板和类型拓展，以增强用户粘性。从整体上看，该系统经过的孵化过程达到了立项初期所设定的学术验证和实际操作输出这两项要求。本文不仅把需求抽象理论转化为实际代码逻辑并生成软件工程实践成果，系统本身也建立起一个待进一步完善以实现工业级商用多方协作模式突破的数字基础。
